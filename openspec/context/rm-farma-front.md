## Architecture patterns

- **Standalone components** for app/route components (`app.config.ts` uses standalone bootstrap).
- **NgRx registration is still module-based**: `RootStoreModule` (`src/app/root-store/root-store.module.ts`) wires `StoreModule.forRoot`, `EffectsModule.forRoot`, and `StoreDevtoolsModule.instrument`, then 18 feature NgModules (`*-store.module.ts`) re-export their slice via `StoreModule.forFeature`. This is a known hybrid state — components are standalone, state is registered via NgModules. Future changes SHOULD keep this pattern until the team commits to a full `provideStore`/`provideEffects` migration (that migration itself is an SDD change).
- **NgRx slice layout** (per feature, e.g. `src/app/root-store/clinic-store/`):
  - `actions.ts`, `state.ts`, `reducer.ts`, `selector.ts`, `effects.ts`, `index.ts`, `<feature>-store.module.ts`
- **Feature pages** live under `src/app/pages/<feature>/` with sibling `*.routes.ts` (lazy-loaded). Sub-features get their own folder (e.g. `check-patient/check-patient-detail`).
- **Domain models** live under `src/app/model/<entity>/` as DTOs (`*-resource.dto.ts`, `*-create-resource.dto.ts`, `*-form-resource.dto.ts`). Pure types, no runtime code.
- **HTTP services** under `src/app/services/` are thin wrappers around `HttpClient` and feed NgRx effects.
- **Cross-cutting utils** under `src/app/utils/`: `interceptors/`, `directives/`, `form-validation/`, `components/` (reusable presentational), `services/` (non-HTTP helpers: layout, excel export, zebra print, crypto local-storage).

## Conventions

- **Standalone components** for app code. NgRx registration via NgModules is the established pattern — keep it consistent until a dedicated migration change.
- **Component selector prefix**: `app` (per `angular.json`).
- **Strict TypeScript and strict Angular templates** are non-negotiable. Do not relax `strictTemplates`, `strictInputAccessModifiers`, or `strictInjectionParameters` without an SDD change.
- **NgRx slice cohesion**: actions / state / reducer / selectors / effects / module grouped per feature folder.
- **Formatting**: 2-space indent, single quotes for `.ts`, UTF-8, final newline (see `.editorconfig`). Prettier 3.4 with defaults — no project-level Prettier config.
- **Budgets**: respect production budgets (500kb / 1mb initial; 2kb / 4kb per-component styles).
- **No ESLint or other linter** is configured — type checking via `ng build` is the main static gate.

## Testing policy

- **Test runner**: Karma 6.4 + Jasmine 5.1, wired via Angular CLI's `karma` builder.
- **Strict TDD Mode**: **ON** — runner available and no `.spec.ts` files exist yet, so all new behavior MUST land with specs (RED → GREEN → REFACTOR). Adding `*.spec.ts` next to source is the expected pattern; `tsconfig.spec.json` already includes `src/**/*.spec.ts`.
- **Spec command** (verify phase): `npm test -- --watch=false --browsers=ChromeHeadless`.
- **Coverage**: `karma-coverage 2.2` is installed; use `ng test --code-coverage`. No coverage threshold enforced today — `coverage_threshold: 0` in `openspec/config.yaml`.
- **Integration / E2E**: not configured. Add Playwright or Cypress only through a dedicated SDD change.

## Known gaps / risks

- **Zero specs**: `src/**/*.spec.ts` count is 0. First changes that introduce behavior will set the testing baseline.
- **No linter**: only `ng build` catches type/template errors. Code style relies on Prettier defaults + review.
- **NgRx hybrid state**: components are standalone, state is registered via NgModules. Keep the pattern consistent across new features until a migration is planned.
- **Coverage threshold is 0**: spec runs are required (strict TDD), but no percentage gate exists. Consider tightening in a future SDD change.
- **No E2E**: regression risk for full flows (login → patient → manufacture) — worth adding Playwright later.
- **Output size**: production budgets are tight for a PrimeNG + FullCalendar + Quill app; new heavy deps need size review.
